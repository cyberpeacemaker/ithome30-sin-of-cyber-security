## 2. 安全測試（Security Testing）

### 滲透測試（Penetration Testing）
模擬駭客攻擊行為，測試系統的實際防禦能力。

- **目的**
  - 找出弱點並驗證可否被利用
  - 測試現有安全控制措施
  - 驗證檢測與回應能量（IR / SIEM）
  - 符合法規或合約需求（如 PCI-DSS）

- **常用工具**
  - Kali Linux, Metasploit, Burp Suite, Nmap, Wireshark

- **基本流程**
  1. 規劃與授權（Scope、ROE、回復計畫）
  2. 偵察（公開資訊、指紋、子域）
  3. 掃描（弱點辨識：自動 + 人工）
  4. 利用（取得初始存取）
  5. 提權與橫向移動（維持存取測試視情況）
  6. 清理（移除測試 artefacts）
  7. 分析與報告（證據、修復建議）

- **測試類型**
  - 黑盒（外部攻擊者視角）  
  - 灰盒（擁有部份資訊）  
  - 白盒（完整資訊，深入測試）

- **風險與注意事項**
  - 可能導致中斷或資料暴露 → 必須有授權、時間窗口、回復機制。
  - 定義清楚測試範圍（不可測試的系統 / 資料 / 用戶群）。

---

## 3. 弱點掃描（Vulnerability Scanning）

自動化比對已知漏洞資料庫與系統配置以找出問題，通常不會利用漏洞。

- **常用工具**
  - Nessus, OpenVAS, Qualys, Rapid7 Nexpose, Microsoft Defender VM

- **流程**
  1. 資產發現（IP、服務、版本）
  2. 掃描識別（CVE 比對、配置錯誤）
  3. 分級與報告（CVSS、建議修補）
  4. 追蹤修補狀態

- **特性**
  - 自動化、高頻執行、非侵入性（理想情況下）
  - 適合作為日常風險管理的一部分

---

## 4. 安全稽核（Security Auditing）

系統性評估組織的資訊安全治理、政策、流程與控管是否符合標準或法規。

- **目標**
  - 驗證合規性（GDPR、HIPAA、PCI-DSS 等）
  - 評估政策、作業流程、控管措施的有效性
  - 提供管理層與第三方可信賴的報告

- **常見框架/標準**
  - ISO/IEC 27001, NIST CSF, COBIT, SOC 2, PCI-DSS, HIPAA

- **主要步驟**
  1. 規劃與範圍定義
  2. 資料蒐集（政策、設定、日誌、存取清單）
  3. 評估（依標準檢查項目）
  4. 報告與風險評估
  5. 改善與追蹤驗證

## 5. 應用程式安全測試（SAST & DAST）

### 靜態應用安全測試（SAST）
- **方式**：分析原始碼、位元碼或二進位檔，無需執行程式。
- **優點**：早期發現邏輯性與語法層級的安全問題。
- **工具**：SonarQube, Checkmarx, Fortify, Veracode, Bandit

### 動態應用安全測試（DAST）
- **方式**：對已部署或執行中的應用程式發動測試（從外部模擬攻擊）。
- **優點**：驗證執行時的安全缺陷（例如會話管理、錯誤處理）。
- **工具**：OWASP ZAP, Burp Suite, Acunetix, Netsparker

### SAST vs DAST（重點比較）
| 比較項目     | SAST                 | DAST                  |
|------------|----------------------|-----------------------|
| 程式碼存取   | 需要（白盒）            | 不需要（黑盒）          |
| 測試階段     | 開發早期               | 部署/測試階段           |
| 偵測內容     | 程式碼缺陷、SQLi、XSS    | 認證繞過、邏輯錯誤、Session |
| 誤報率      | 較高（需人工判讀）         | 較低                   |
| 適合對象     | 開發團隊               | 測試/安全團隊            |

**建議**：在 CI/CD 中整合 SAST（PR/CI 階段），在測試環境執行 DAST；兩者互補，可有效降低風險。

## 6. 黑盒與白盒測試（Black-box / White-box）

| 測試類型   | 定義                                   | 適用情境         |
|----------|--------------------------------------|--------------|
| 黑盒測試   | 基於介面與需求，只驗證輸入/輸出               | 功能測試、模糊測試、DAST |
| 白盒測試   | 基於內部結構與程式碼，針對路徑/分支編寫測試        | 單元/集成測試、SAST    |

### 黑盒常用技術
- 模糊測試（Fuzzing）：隨機或變異輸入發現崩潰/例外
- 等價類劃分（Equivalence Partitioning）
- 邊界值分析（Boundary Value Analysis）

### 白盒常用技術
- 單元測試（Unit Testing）
- 覆蓋率分析（Branch/Condition Coverage）
- 代碼路徑分析（流程/控制流）

## 7. 源碼檢視與靜態分析（Code Review & Static Analysis）

- **源碼檢視**：人工或同儕審查程式碼（邏輯錯誤、可讀性、資安疑慮）
- **靜態分析**：自動化工具檢查風險（記憶體洩漏、未初始化使用、潛在注入點）

**工具例子**：ESLint, SonarQube, Bandit, Flake8, Fortify SCA

**建議做法**：
- PR 時進行自動化 SAST 並搭配人工 code review
- 建立檢視清單（安全 checklists）以統一審查重點
- 定期更新規則與 false-positive 清單

## 8. 比較總表：滲透測試 vs 弱點掃描 vs 安全稽核

| 項目             | 滲透測試 (Pen Test)           | 弱點掃描 (Vulnerability Scan) | 安全稽核 (Security Audit)        |
|----------------|------------------------------|-------------------------------|----------------------------------|
| 目的             | 模擬攻擊、發現可利用弱點           | 自動發現已知漏洞與錯誤設定         | 驗證政策、控管、合規性               |
| 是否模擬攻擊        | ✅ 是                          | ❌ 否                          | ❌ 否                           |
| 是否利用漏洞        | ✅ 有（可選擇測試深度）             | ❌ 否                          | ❌ 否                           |
| 風險             | 中高（可能中斷、影響系統）           | 低（非侵入）                    | 低（文件與流程檢查為主）             |
| 頻率             | 定期（季/半年/重大變更）            | 高頻（每日/週或每次部署）         | 定期（年度或合規需求）               |
| 輸出             | 攻擊路徑、利用證據、修補優先清單        | 漏洞清單、CVSS、修補建議            | 稽核報告、合規狀態、改進計畫            |
| 適用對象          | 安全團隊 / 第三方測試者              | IT/運維 / 安全團隊                | 風險/合規/管理層、第三方稽核            |

## 9. 實務範本（Templates）

### A. 滲透測試簡要範圍範本（Scope）
- 測試名稱：＿＿＿＿＿＿＿＿＿＿
- 測試類型：黑盒 / 灰盒 / 白盒
- 測試範圍：域名、IP、API 路徑、子系統（列出明確清單）
- 不可觸碰資產（禁止測試）：付款系統、實時交易系統、客戶個人資料庫
- 時間窗：YYYY-MM-DD hh:mm ~ hh:mm
- 授權聯絡人：姓名、電話、Email
- 回復計畫：服務中斷回復流程與聯絡人
- 測試成果交付：報告格式、交付時間、會議日程

### B. 滲透測試報告大綱（簡要）
1. 摘要（高階發現、業務風險）
2. 測試目標與範圍
3. 測試方法與工具
4. 重大發現（分級：Critical / High / Medium / Low）
   - 發現描述
   - 證據（截圖、POC）
   - 影響評估（業務、合規）
   - 修復建議（步驟與優先順序）
5. 測試過程記錄（重要活動時間線）
6. 附錄（掃描輸出、CVE 列表、變更紀錄）

### C. 基本安全檢查表（Short Checklist）
- [ ] 使用強密碼策略 & MFA（管理帳號）
- [ ] 所有伺服器與依賴套件打補丁
- [ ] 最小權限原則（Least Privilege）有落實
- [ ] 備份與恢復流程測試過（可用性驗證）
- [ ] 機敏資料加密（傳輸與靜態）
- [ ] 日誌與監控（SIEM）設定與告警
- [ ] 公網服務有 WAF 或速率限制保護
- [ ] 第三方套件/庫有定期掃描與更新流程

## 10. 建議整合到 DevSecOps（實務做法）

- 在 CI/CD pipeline 中：
  - PR / CI：自動執行 SAST、靜態檢查與單元測試（針對阻斷規則）
  - Nightly 或 Merge 後：執行依賴檢查、SCA（Software Composition Analysis）
  - 測試環境：自動執行 DAST 與 E2E 安全場景測試
- 基礎設施即代碼（IaC）：
  - 使用 IaC 靜態檢查工具（如 Terraform/CloudFormation scanners）
- 監控與回應：
  - 將滲透測試中發現的攻擊向量加入偵測情境（SIEM 用到）
- 風險管理：
  - 建立「發現→評估→修補→驗證」的 SLA 與流程（依漏洞嚴重度）
- 教育與流程：
  - 定期安全訓練與紅藍隊演練（Tabletop 演練）
  - 建立觀測指標（MTTR、平均修補時間、漏洞密度）

## 結語
- 建議採用「多層次」的防護與測試策略：SAST（開發）→ DAST（測試）→ 弱點掃描（運維）→ 滲透測試（紅隊/外部評估）→ 安全稽核（合規/治理）。  
- 對於任何具破壞風險的測試（如滲透測試或高強度 fuzzing），務必先取得書面授權、定義 ROE（Rules of Engagement），並準備回復計畫與聯絡流程。
