# 【15】資安的原罪 ch.3-2.b 社交工程

本中見證實際發生最常見的攻擊。

---

## 什麼是社交工程?

**社交工程**（Social Engineering）是指透過操縱或欺騙人們，使其洩露機密資訊或執行可能危害資安的行為。這種攻擊方式**利用的是人類心理**，而不是技術漏洞。

## 社交工程有哪些常見類型？

### 📨 1. **網路釣魚（Phishing）**

**定義與手法**

* 攻擊者透過電子郵件、即時通訊、社交平台或假網站，偽裝成可信來源（銀行、同事、廠商、系統通知等），誘使使用者點擊連結、下載附件或輸入帳號密碼。
* 常見技巧：偽造寄件人、使用相似域名（ex: “paypa1.com”）、在郵件內嵌急迫語句（「24 小時內…」）或財務、合規類緊急通知。

**常見變種（補充）**

* *定向釣魚（Spear phishing）*：以**特定**個人或組織為目標，內容**高度客製**（引用內部資料、職務、近期專案），成功率高。
* *高階釣魚 / 鯨魚攻擊（Whaling）*：針對**高階主管（CEO/CFO）或重大決策者**，常偽造法律、財務或併購相關文件。
* *簡訊釣魚（SMS phishing / Smishing）*：透過 **SMS 或即時訊息**發送惡意連結或驗證碼催促。
* *克隆釣魚（Clone phishing）*：攻擊者先取得**合法郵件範本**，替換附件或連結為惡意版本，再用看似同樣的發信地址寄回給原收件人。
* *域名導流（Pharming）*：透過 DNS 中毒或其他方式，把原本正確網址導向惡意網站（使用者輸入**正確網址卻被導到假站**）。

### ☎️ 2. **語音釣魚（Vishing）**

**定義與手法**

* 攻擊者以電話或語音訊息假冒內部單位（IT、HR、財務）或外部機構（銀行、政府單位），用口語說服受害者交出帳號、密碼、驗證碼或進行轉帳。
* 技術上常配合 **來電顯示偽造（caller ID spoofing）**，讓來電看起來來自可信號碼。

**AI & 深度偽造（新趨勢）**

* 攻擊者利用 AI 合成目標對象的聲音（deepfake voice）或生成**極為擬真的語音訊息**，使受害者更易相信來電者是熟人或主管。
* 也會把 AI 用於自動化大規模撥號與腳本調整，根據受害者回應即時改變社交策略。

企業須正視語音詐騙的資安危機[^4
]北韓駭客BlueNoroff在Zoom視訊利用AI深偽技術對企業員工網釣[^3]

### 🕵️ 3. **假借身份（Pretexting）**

* 攻擊者編造合理故事或角色，以套取資訊（例如假扮人資或廠商）

### 🏢 4. **誘餌攻擊（Baiting）**

* 實體誘餌：在公司放置標有「內部資料」的 USB，或把惡意裝置留在停車場/茶水間；數位誘餌：假工作、免費軟體下載。

### 👨‍💼 5. **冒充 / 尾隨（Impersonation / Tailgating）**

* 假扮員工、快遞或維修人員，實體闖入安全區域

> *「社交工程攻擊者不需要管理員權限，他們只需要一個好故事和一點自信。」*
中國駭客行冒充美國國會議員，活動鎖定參與美中關係、國際貿易與經濟政策的美國政府、智庫與學術單位。.[^10]

### 📨 假寄信人（透過「回信機制」的攻擊）— 補充說明


## 1) 攻擊手法概覽（攻擊者怎麼做）

* **顯示名稱偽造（Display name spoofing）**：在郵件顯示名稱寫上可信單位或人名（例如「HR‑Admin」），但實際電子郵件地址是惡意域名或外部帳號。使用者常只看顯示名稱就回信。
* **Reply‑To 操作（Reply‑To manipulation）**：寄件者在郵件標頭內設定 `Reply‑To` 為攻擊者地址。使用者按「回覆」時，郵件會發送到攻擊者而非顯示的寄件人。
* **Return‑Path / Mail From 偽造**：發送方（SMTP envelope）被偽造，導致回覆/退信流向惡意地址。
* **回信鏈插入（Reply chain injection）**：在原始郵件的引用區或尾端插入惡意 `From:` 或 `Reply-To:` 行，欺騙收件者直接回覆那個插入的地址（尤其在轉寄/回覆大量郵件時容易發生）。
* **帳號被入侵或轉寄規則被設定**：真正的內部帳號被攻破或被設置自動轉寄，把回覆導到外部。
* **顯示名稱同形字（homograph / Unicode spoofing）**：域名用看似相同但不同 Unicode 字元（例：payрal.com，用西里爾字母 р 代替拉丁 p）讓使用者誤判。
* **郵件中嵌入回覆按鈕或指示**：例如「請按此回覆以確認」，但那按鈕其實觸發回覆到攻擊者的郵件地址。

## 2) 範例示意（簡短示例，方便放投影片）

範例 A — Reply‑To 操作：

```
From: "IT Support" <it-support@yourcompany.com>
Reply‑To: it.support.verify@evil.com
Subject: 請確認你的帳號
```

使用者看到「From: IT Support」，按回覆→實際收件人是 `it.support.verify@evil.com`。

範例 B — 回信鏈插入（轉寄／回覆大量收件人）

```
--- 原始訊息 ---
From: alice@partner.com
To: team@yourcompany.com
...
--- 以下為插入內容 ---
From: support@evil.com
Subject: Re: 請確認...
```
## 3) 如何辨識（紅旗清單 — 使用者層級）

* 在回覆前，先看郵件的「收件人（To:)」或郵件回覆畫面裡的實際「收件人」欄位：它是否與你預期的一致？
* 檢查「Reply‑To」欄位：若有該欄且地址與 From 不同，特別小心。
* 檢視完整寄件人電子郵件地址（不要只看顯示名稱）。
* 留意域名同形字（例如含奇怪字元或長度不合常理）。
* 如果郵件是轉寄或回覆鏈，檢查引用區是否被插入新寄件者或新的回覆指示。
* 若郵件要求敏感資訊或匯款，堅持用獨立渠道（電話、公司內部 IM）再次確認。
* 注意郵件標題或內文中包含「請直接回覆此郵件至…」或「使用此按鈕回覆」這類指示，先檢查按鈕的實際 target。


### 📞 簡單介紹：Caller ID Spoofing（來電顯示偽造）

**什麼是它？**
Caller ID spoofing 指攻擊者改寫或偽造來電顯示（來電號碼/來電者名稱），讓接聽者看到的是可信的號碼（例如銀行、公司內線、主管手機），其實真正的發話方是惡意者。這常被用於詐騙電話（vishing），提高受害者信任以取得敏感資訊或授權匯款。

**常見做法（簡述）**

* 使用網路電話服務（VoIP）或公開 API，將顯示號碼設定成任意值。
* 利用電話中繼或國際閘道改寫來電顯示。
* 結合社交工程（例如假扮銀行或 IT）與偽造號碼，讓受害者降低警覺。

**典型攻擊場景**

* 偽裝銀行來電，通知「帳戶有異常交易，請提供 OTP」→ 詐取一次性密碼。
* 偽裝公司高層要求緊急匯款→ 社內財務照辦。
* 偽裝公家機關要求提供個資或付款資訊。

## 社交工程帶來什麼威脅

* Works even when **systems are technically secure**
* Targets the **weakest link**: human trust
* Can be the **first step** in a much larger breach (e.g. phishing to get credentials ➝ lateral movement)

> *“It’s easier to trick a person than to hack a machine.”*
